# 2025-10-03
https://layerx.connpass.com/event/367170/
AI Coding Meetup #3 AI時代の開発スピードと品質のバランス術

output は増えたけどアウトカムは？  

開発フロー全体で見ると歪みが溜まっていってる

review, test, QA はタスク量が増えている  
バグ・手戻りが多いと価値提供や事業成長が伸びない

結論：アウトカムは増えてない by `naoya_ito`

生産性が上がった -> speed が上がった -> 楽になった、と言っている（エンジニア視点）

code 量が増えていてメンテが大変になりそう  
ここどうなってる？ -> はて？みたいなパターンが増えたような気がする  
楽なほうを選ぶのは人間の心理よね  

ソフトウェアでも品質の高さを求める部分とそうでない部分がある

UI の提供には AI agent が向いている。多少コードの品質が下がっても。  

空いた時間を本質的な仕事に使おう？ってならないよね。QOL は上がったけど会社のアウトカムは増えてない  
倍働いたから倍給料もらえるわけじゃない（真理）  

まだ評価が早すぎる。開発のスピードが速くなったことは大した問題にならない  
現代はサービスを提供するのに時間がかかっている。なぜ？質が上がっているから。  

これから何年かしてユーザーがサービスに求める質が上がったときに、これは AI 使わないと無理だねってなったらはじめて評価できる  

小規模な案件は明らかに減ってる。発注者が自分で作れるから。  
発注者側がプログラミングを理解すればハッピーになりそう。  
個人としては裾野が広がり全体の質が下がる。このくらいで良いじゃない。  
企業としてはそれよりも質の高いものを提供しないといけない。ハードモードやな  

designer が作るようになったけどプルリクの変更量が大きかったりすると品質を担保するのが難しい  

モデリング が大事！それをきちんとトレースしたコードになっているかどうかが大事

test はゼロから書かせると良くない。例を提示してそれを参考にしてもらうと良い  
問題は誤った安心感が生まれてしまうこと。カバレッジは増えたが。。。コード量やモックオブジェクトは増えたがメンテが大変になる  

ユニットテストをうまく書けばいい。人間側がどううまいお手本を見せるか  
わかってる人じゃないと指示できない。  

逆に全くわからない時はゼロから AI とやってみるのもいい。  

口頭で生まれたドメインに対する共通認識が即AI に入るわけではない。  

はじめはみんな手作業でやったりしてるが、その間はまだモデルが出来上がっているわけではない。  
そのプロセスを経てシステムへと落とし込むが、システムに近くなって初めて AI が役にたつ  

AI と一緒にモデリングしていくと平均に回帰していく。それだと競争優位性を出せない  
むしろ人間が AI から離れていかないといけない。一番最初にAI に叩き台を出させるのはいいが  

AI がかけるということは他の人も書けるということ  

AI は引き算が苦手
